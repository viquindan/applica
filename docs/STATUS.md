## SSE para refresco en tiempo real del dashboard web (2026-07-22)
Pedido explícito del usuario, confirmado que entraba después de cerrar el noVNC. Reemplaza dos loops de polling client-side por una sola conexión SSE por pestaña (`src/app/api/events/route.ts`): el `router.refresh()` cada 4s que hacía `useApplicationActions` mientras había una aplicación `approved` enviándose, y el `fetch('/api/search/status')` cada 2s de `useSearchEngine` mientras corría una búsqueda. El endpoint hace su propio poll a la DB cada 1.5s (por usuario, no por pestaña) y solo empuja un frame SSE cuando el fingerprint del estado realmente cambió - una pestaña sin nada nuevo no genera tráfico más allá del heartbeat.
Decisión de alcance explícita: NO se conectó a los ~9 puntos de transición de status en `worker.ts` (zona sensible de apply-engine, `docs/APPLY-ENGINE.md`) - pollear desde el propio route handler no requirió tocar ese código a cambio de hasta 1.5s de latencia en vez de push instantáneo real vía Postgres LISTEN/NOTIFY, que habría necesitado instrumentar cada `db.update(applications).set({status: ...})` disperso en un archivo grande y activo. Si en el futuro la latencia de 1.5s resulta insuficiente, la migración a LISTEN/NOTIFY queda como mejora incremental, no un rediseño.
`useLiveEvents` (hook cliente, `src/app/(dashboard)/applications/useLiveEvents.ts`) se cablea una sola vez por página (Feed/Pendientes/Apps) para evitar conexiones SSE duplicadas cuando una página usa tanto `useApplicationActions` como `useSearchEngine` (Feed). Verificado en producción con un token real: la conexión abre, autentica, y empuja `applications_changed`/`search_progress` en el primer tick. `npm test` 78/78, `tsc --noEmit` limpio, `npm run build` limpio.

## Ver y ayudar en vivo al flujo asistido (noVNC), pool de 10 sesiones concurrentes, en producción (2026-07-22)
Plan aprobado explícitamente por el usuario (`C:\Users\lidic\.claude\plans\generic-beaming-donut.md`): cuando el flujo asistido se congela esperando un captcha, el navegador headful vivía en un framebuffer invisible (Xvfb `:99`) sin forma de que el usuario lo viera o interviniera dentro de los 15 minutos de espera. Implementado noVNC (cliente VNC en el navegador, sin instalar nada) con acceso por sesión, no por sistema, requisito explícito del usuario ("si hay 1000 usuarios, deberian ser posibles 1000 ventanas"):
- **Pool de 10 displays aislados** (`Xvfb :100`-`:109` + `x11vnc` + `websockify`, 30 servicios systemd templados `*-pool@N`, puertos VNC 5900-5909/websockify 6080-6089) - cada sesión asistida toma un slot libre (`ASSISTED_APPLY_POOL_SIZE=10`), `boss.work('assisted_apply', { localConcurrency: 10 }, ...)` reemplaza el default de 1 concurrente que era el techo real del sistema entero (no por usuario). `applications.assistedSessionStartedAt`/`assistedSessionPoolIndex` (nuevo, nullable) es la señal en DB de qué slot está usando cada aplicación, porque el pool en memoria vive en el proceso worker y la API corre en un proceso PM2 aparte.
- **Token firmado de un solo uso** (`liveSessionToken.ts`, HMAC + 5 min TTL) minteado por `GET /api/applications/[id]/live-session` (valida dueño + sesión viva) y verificado por nginx vía `auth_request` contra `GET /api/internal/verify-live-session` antes de dejar pasar el proxy al websockify del slot correspondiente. Push notification (`sendPushToUser`) avisa al usuario apenas aparece un captcha real. Botones "Ayudar ahora (en vivo)" en Pendientes, web y mobile (WebView dedicado en mobile, pestaña nueva en web).
- **3 bugs reales encontrados y arreglados durante la verificación end-to-end** (no en el diseño, en la integración real nginx+Next.js+VPS):
  1. **`AUTH_SECRET` en `.env.local` tenía un byte de retorno de carro (`\r`) incrustado DENTRO del valor** (`AUTH_SECRET="...TU0=<CR>"` - contaminación de line-endings de Windows), invisible en cualquier editor/grep normal. Distintos procesos lo parseaban de forma inconsistente, así que cada token de prueba que yo minteaba fallaba la verificación aunque la lógica de firma/verificación en sí era correcta - esto fue la causa real de horas de debugging que parecían apuntar a un bug de nginx. Arreglado limpiando el archivo compartido (`/var/www/applica-shared/.env.local`) y con `pm2 startOrReload` para que ambos procesos recarguen el secreto correcto (efecto secundario esperado y aceptado: cierra sesión a todo usuario logueado en ese momento, web/mobile/extensión).
  2. **`$arg_token` no se interpola dentro del argumento propio de la directiva `auth_request` en este nginx** (confirmado con `error_log debug`: pasaba el texto literal `"$arg_token"` en vez de expandirlo a su valor real). Reemplazado por el patrón robusto estándar (`auth_request` a una URI fija sin variables + `proxy_set_header X-Original-URI $request_uri` + el propio `verify-live-session/route.ts` lee el token de ese header en vez de su propio query string) - `$request_uri` sí se resuelve de forma confiable dentro de una subrequest, a diferencia de `$arg_*`.
  3. **El token solo se validaba contra el estado vivo en DB, nunca contra QUÉ índice de pool pedía la URL** - un token válido para el slot 0 también pasaba `auth_request` contra la URL del slot 1 con solo cambiar el dígito, dejando ver la sesión de otro usuario. `verify-live-session/route.ts` ahora cruza el índice del path (`/assisted-view/<i>/websockify`) contra el `poolIndex` firmado en el token antes de aprobar.
- Verificado en producción con una aplicación de prueba real (Webflow, id `41d50781-...`, cancelada limpia sin enviar): token válido + slot correcto pasa el `auth_request` y llega a websockify; token válido + slot equivocado → 403; token inválido/ausente → 401. `npm test` y `tsc --noEmit` limpios (toca `applications` schema y zona de apply-engine, regla dura del proyecto).
- Pendiente (documentado en el plan, no bloqueante): prueba de concurrencia real con 2-3 sesiones simultáneas desde cuentas distintas; SSE para refresco en tiempo real del dashboard web (confirmado explícitamente que entra después de esto). ## Causa raíz real de "aplicaciones que no se envían" + quitado el cap de 200 candidatos + bug de deploy sin build atómico (2026-07-21)
El usuario reportó 3 swipes positivos que no aparecían ni en Aplicaciones ni en Pendientes. Diagnóstico contra logs reales de producción (nginx access log + PM2 logs de `applica-web`/`applica-worker`, la consulta SQL directa a la DB de producción está bloqueada por el sistema de permisos):
- **Causa raíz confirmada por nginx**: cada swipe llamaba `POST /api/applications/[id]/action` con `action:'approve'` y el servidor respondía **409** (`La aplicación todavía tiene datos obligatorios pendientes.` + lista de `blockers`) - el intento silencioso/headless rechaza correctamente enviar un formulario con campos requeridos que la IA no pudo responder con confianza (`formPreview.blockers`, snapshot de `prepare_application_materials`).
- **El bug real no era el 409 en sí (correcto, per `docs/APPLY-ENGINE.md` §1: dato genuinamente desconocido = razón válida para intervención humana), sino que nada escalaba automáticamente al flujo asistido (navegador real) como sí hace el worker con un captcha.** `useApplicationActions` (mobile) no tenía NINGÚN manejo de error en `archive`/`discard`/`markApplied`/`cancelAssisted`/`applyAnyway`/`answerBlockers`, y `sendApprove`/`sendAssisted` sí lo capturaban pero solo se mostraba en la pantalla de detalle, nunca en el Feed - un swipe fallido se veía IDÉNTICO a uno exitoso (la tarjeta volaba y no pasaba nada más). El resto de mutaciones ahora sí capturan error (para otras pantallas), pero el Feed deliberadamente no lo muestra.
- **El "muro de 200" del funnel no era un bug de datos sino un límite de performance real** (`limit: 200` en `worker.ts::search_vacancies`, sobre el pool ya filtrado por rol+región+remoto): antes de puntuar de verdad, el pre-filtro (`filterRankLimit` en `atsSearchHelpers.ts`) rankea por un heurístico rápido (rol+geografía+antigüedad+salario) y se queda con el top-200 - cualquier candidato que hubiera puntuado bien en el scorer real pero quedara fuera de ese top-200 nunca se evaluaba, run tras run. Quitado por decisión explícita del usuario ("dale todas las oportunidades posibles"): el límite subió a un valor efectivamente sin techo (999999, mismo patrón que ya usaba `roleOnlyPool` en el mismo archivo para telemetría). Verificado que el costo por candidato es barato (eligibility/fitScorer son JS puro; `maybeSemanticAdjust` solo llama a la API de embeddings para candidatos borderline Y solo si `ENABLE_SEMANTIC_RERANK=true`) - el cambio hace la búsqueda más lenta, no más cara en IA.
- **Hallazgo de infraestructura, no arreglado todavía (requiere decisión del usuario)**: el pipeline de deploy (`.github/workflows/deploy.yml`) corre `npm run build` **en el mismo directorio `.next` que `applica-web` está sirviendo en vivo**, sin build atómico/directorio temporal - confirmado en logs reales (`Cannot find module .../setup-node-env.external.js`, `Could not find a production build`, `Failed to find Server Action "x"`, 17 ocurrencias en los últimos 500 logs) que ocurren en la ventana de cada deploy mientras el proceso viejo sigue sirviendo tráfico contra un `.next` a medio escribir. Con varios deploys en el mismo día (como hoy) la probabilidad de que una request real de un usuario caiga en esa ventana no es despreciable.
